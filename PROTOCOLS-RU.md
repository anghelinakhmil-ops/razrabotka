#### 3A. ЦИКЛ TASK-VERIFY-FIX (КРИТИЧЕСКИ ВАЖЕН!)

**ЗОЛОТОЕ ПРАВИЛО пошагового выполнения:**

```markdown
════════════════════════════════════════════════
ЦИКЛ TASK-VERIFY-FIX — ОБЯЗАТЕЛЕН!
════════════════════════════════════════════════

ДЛЯ КАЖДОГО ШАГА из плана:

STEP X: [Действие]
  1. Помечаю задачу как "in_progress"
  2. Выполняю шаг полностью
  3. Помечаю задачу как "completed"
  4. ПРОВЕРКА НЕМЕДЛЕННО! (НЕ продолжаю к следующему!)
     → npm run build ИЛИ npm test ИЛИ lint
  5. Если ОШИБКИ → FIX немедленно → повторная проверка
  6. Если УСПЕХ → ТЕПЕРЬ перехожу к STEP X+1

ЗАПРЕЩЕНО:
- 3+ шагов подряд без проверки
- "Оставлю проверку на конец"
- Продолжать с активными ошибками

ПРИМЕР ПРАВИЛЬНОГО ВЫПОЛНЕНИЯ:
STEP 1: Создать компонент → выполнено
  → npm run build ← НЕМЕДЛЕННО!
  → 2 ошибки → FIX → build → УСПЕХ
STEP 2: Добавить тесты → выполнено
  → npm test ← НЕМЕДЛЕННО!
  → 1 тест упал → FIX → test → УСПЕХ
STEP 3: Обновить документацию → выполнено
  → (пропуск проверки — только документация)

ПРЕИМУЩЕСТВА:
- Мгновенные исправления, пока ошибка свежая
- Ноль накопленных ошибок в конце
- 50-70% экономия времени
- Профессиональный рабочий процесс

════════════════════════════════════════════════
```

```markdown

═══════════════════════════════════════════════

ПРАВИЛА КОНСИСТЕНТНОСТИ (АВТОМАТИЧЕСКИЕ IF-THEN):

1. ПРАВИЛО: Новые файлы vs Structure Protocol
   IF "Нужны новые файлы: ДА"
   THEN "02-STRUCTURE.md" ДОЛЖЕН быть "ДА" (АКТИВЕН!)
   ИНАЧЕ → ПОЛНАЯ БЛОКИРОВКА! Обнаружено противоречие!

2. ПРАВИЛО: UI-изменения vs UI Verification Protocol
   IF "Нужны UI-изменения: ДА"
   THEN "03-UI-VERIFICATION.md" ДОЛЖЕН быть "ДА" (АКТИВЕН!)
   ИНАЧЕ → ПОЛНАЯ БЛОКИРОВКА! Обнаружено противоречие!

   ПРИМЕЧАНИЕ: "UI-изменения" = ЛЮБОЕ визуальное изменение в UI:
   - Индикаторы загрузки (CircularProgressIndicator)
   - Изменения ListView (пустой → заполненный)
   - Новые кнопки
   - Новые экраны
   - Видимые изменения текста
   - Изменения иконок
   - Изменения layout
   - Изменения цветов
   - Новые анимации
   НЕ СУЩЕСТВУЕТ "мелких UI-изменений", которые могут обойти протокол!

3. ПРАВИЛО: Breaking changes vs Breaking Changes Protocol
   IF задача изменяет существующие API (endpoints, модели, контракты)
   THEN "15-BREAKING-CHANGES.md" ДОЛЖЕН быть активирован!
   ИНАЧЕ → ПОЛНАЯ БЛОКИРОВКА!

4. ПРАВИЛО: Frontend UI vs WEB-TEST Protocol
   IF задача изменяет UI фронтенда + нужна визуальная верификация
   THEN "08-WEB-TEST.md" должен быть упомянут
   ИНАЧЕ → ПРЕДУПРЕЖДЕНИЕ (но не блокировка)

5. ПРАВИЛО: Сложность vs TodoWrite
   IF Сложность = "Средняя" ИЛИ "Сложная" ИЛИ 3+ шагов
   THEN TodoWrite ДОЛЖЕН быть создан в ПЕРВОМ ответе
   ИНАЧЕ → ПОЛНАЯ БЛОКИРОВКА! (Протокол TodoWrite MANDATORY забыт!)

6. ПРАВИЛО: OPA всегда активен
   IF любая задача с кодом/изменениями
   THEN "07-OPA.md" ДОЛЖЕН быть АКТИВЕН (ВСЕГДА!)
   ИНАЧЕ → ПОЛНАЯ БЛОКИРОВКА! OPA ОБЯЗАТЕЛЕН!

7. ПРАВИЛО: Git commit всегда активен
   IF любая завершённая задача
   THEN "10-GIT-COMMITS.md" ДОЛЖЕН быть активирован
   ИНАЧЕ → ПОЛНАЯ БЛОКИРОВКА! Стандарты коммитов забыты!
```

##### Когда добавляю что-то НОВОЕ (инструмент/фича):

**1. СТОП НЕМЕДЛЕННО**

Если не существует в CLAUDE.md или текущей структуре:

```
РЕКОМЕНДАЦИЯ СТРУКТУРЫ для [НАЗВАНИЕ_ФИЧИ]:

Предлагаемое расположение:
  - Папка: /path/to/appropriate/location
  - Файл: [имя_файла].ext

Почему здесь:
  - [Причина 1: Например, "Это независимый инструмент"]
  - [Причина 2: Например, "Не зависит от других компонентов"]
  - [Причина 3: Например, "Легко тестировать изолированно"]

Альтернативы:
  - Вариант A: /path/alternative1 (если X)
  - Вариант B: /path/alternative2 (если Y)
```

**3. ПРОШУ ОДОБРЕНИЕ**

```
Куда хотите поместить [НАЗВАНИЕ_ФИЧИ]?
  A) Моя рекомендация: [путь]
  B) Другое расположение: [указать]
```

**ЗОЛОТОЕ ПРАВИЛО:**
- **ВСЕГДА** отдельный файл/папка

- Полная изоляция от другой функциональности

```
НЕПРАВИЛЬНО:

ПРАВИЛЬНО:
src/
  utils.py              # Существующий код не тронут!
    - existing_function()
  tools/                # НОВАЯ папка
    new_tool.py         # НОВЫЙ отдельный файл
      - NEW_TOOL()
```

```markdown
### СТРУКТУРА ПРОЕКТА
```

## ПРОТОКОЛ 06: УПРАВЛЕНИЕ ФАЗАМИ — Разработка по фазам

**Важность:** CRITICAL

## ПРОТОКОЛ 07: OPA — Omniscient Production Assurance

**Что делает OPA:** Полный и универсальный протокол обеспечения качества для ЛЮБОГО типа проекта.

**Статус:** ОБЯЗАТЕЛЕН — ВСЕГДА перед commit
**Важность:** CRITICAL

---

### ПРОВЕРКИ OPA (адаптируемые к типу проекта):

#### 1. **Верификация build** (КРИТИЧЕСКИ ВАЖНО — НУЛЕВАЯ ТОЛЕРАНТНОСТЬ)
- **НОЛЬ допустимых проблем** (ошибки ИЛИ предупреждения) — ВСЕ должны быть исправлены!
- **НЕ существует "допустимых предыдущих проблем"** — исправь ВСЕ или НЕ коммить!
- Все warnings обрабатываются как БЛОКИРУЮЩИЕ ошибки
- Build должен пройти 100% чисто (0 errors, 0 warnings, 0 info)
- Поддерживает: TypeScript, JavaScript, Python, PHP, Ruby и т.д.
- **Пример:** `npm run build && npm run lint` ДОЛЖЕН пройти без ошибок!

#### 2. **Проверки качества кода** (НУЛЕВАЯ ТОЛЕРАНТНОСТЬ)
- Никаких `TODO`, `FIXME`, `HACK` в коде
- Обнаружение антипаттернов специфичных для языка:
  - TypeScript: `any` type, `@ts-ignore`, `@ts-nocheck`
  - JavaScript: `console.log` в production, `var` вместо `const/let`
  - Python: bare except, mutable defaults
  - CSS: `!important` без обоснования, inline styles
- Ноль захардкоженных секретов

## ПРОТОКОЛ 09: CI/CD — Удалённый Quality Gate и Deployment

**Статус:** ОБЯЗАТЕЛЕН для проектов с CI/CD (GitLab, GitHub Actions, etc.)
**Важность:** CRITICAL

---

### КОМПОНЕНТЫ CI/CD:

#### 1. **Pre-push Hook** — Локальные проверки ПЕРЕД push
- OPA-валидация, формат commit message, конвенция именования веток, обнаружение секретов
#### 2. **CI/CD Pipeline** — Многоэтапная автоматизация
- Этапы: validate → build → test → quality → deploy
- Параллельные задачи, кэширование зависимостей

#### 3. **Стандарты Merge/Pull Request**
- Шаблоны MR/PR, руководство по ревью, правила одобрения

#### 4. **Стратегии Deployment**
- Staging → Production (всегда!), canary/blue-green deployment, автоматический rollback

#### 5. **Защищённые ветки**
- Правила защиты, обязательный pipeline success, обязательное одобрение
- Force push отключён

---

### PRE-PUSH HOOK

```bash
cp quality-system/examples/pre-push .git/hooks/pre-push
chmod +x .git/hooks/pre-push
```

Проверяет: (1) OPA-валидация, (2) формат commit message (Conventional Commits), (3) конвенция именования веток, (4) сканирование секретов.

### ЭТАПЫ PIPELINE (5 стадий)

```
validate → build → test → quality → deploy

Этап 1: VALIDATE  — Линтинг, форматирование, сканирование секретов
Этап 2: BUILD     — Компиляция, артефакты сборки
Этап 3: TEST      — Unit-тесты, интеграционные тесты, E2E
Этап 4: QUALITY   — OPA-валидация (ОБЯЗАТЕЛЬНО!), анализ качества кода
Этап 5: DEPLOY    — Staging (авто/ручной) → Production (только ручной)
```

### КОНВЕНЦИЯ ИМЕНОВАНИЯ ВЕТОК

```
feature/*  — Новые фичи (feature/user-auth, feature/dashboard-charts)
bugfix/*   — Исправления багов (bugfix/login-crash, bugfix/api-timeout)
hotfix/*   — Экстренные фиксы production (hotfix/security-patch)
release/*  — Подготовка релизов (release/v2.1.0)
main       — Production (защищённая, push запрещён, только через MR)
develop    — Staging (защищённая, deploy автоматический)
```

### ЧЕКЛИСТ MERGE REQUEST (краткий)

```
□ OPA-валидация пройдена локально (./OPA.sh → 100% SUCCESS)
□ Тесты добавлены/обновлены, все проходят
□ Нет console.log/debug/секретов в коде
□ Pipeline пройден (все этапы зелёные)
□ Self-review завершён, документация обновлена
□ Breaking changes задокументированы (если есть)
```

### ПОРЯДОК DEPLOYMENT

```
1. Push в feature/* → Pipeline (validate+build+test+quality)
2. MR feature/* → develop → Merge → Авто-deploy на STAGING
3. Тестирование в staging-окружении
4. MR develop → main → Merge → Ручной deploy на PRODUCTION
5. Мониторинг работоспособности после деплоя
```

### ОТКАТ (ROLLBACK)

При проблемах на production — немедленный откат к предыдущей стабильной версии.
Задача `rollback:production` в pipeline (ручной запуск). Подробности: Протокол 18-ROLLBACK.

### PRE-PUSH HOOK

Устанавливается в `.git/hooks/pre-push`. Проверяет: (1) OPA, (2) формат commit, (3) имя ветки, (4) отсутствие секретов. При провале — push блокируется.

```

```

## 🎯 ПОЛНЫЙ WORKFLOW ОТЛАДКИ:

### ФАЗА 1: ВОСПРОИЗВЕДЕНИЕ (КРИТИЧЕСКИ!)

**Правило #1:** НИКОГДА не пытайся исправить баг, который не можешь воспроизвести!

```markdown
## ЧЕКЛИСТ ВОСПРОИЗВЕДЕНИЯ:

1. ✅ **Точные шаги** — Конкретные шаги для воспроизведения
2. ✅ **Окружение** — ОС, браузер, устройство, версии
3. ✅ **Входные данные** — Точные данные/параметры, вызывающие баг
4. ✅ **Ожидаемое поведение** — Что должно происходить
5. ✅ **Фактическое поведение** — Что происходит на самом деле
6. ✅ **Скриншоты/видео** — Визуальные доказательства
7. ✅ **Логи/ошибки** — Вывод консоли, сообщения об ошибках

9. ✅ **Минимальное воспроизведение** — Простейший случай с багом
10. ✅ **Тест-кейс** — Автоматический тест, который падает
```

### ФАЗА 2: ДИАГНОСТИКА (Протокол WEI)

**Обязательно:** Выполни WEI Protocol (см. 04-WEI.md) для полной диагностики!

**Вопросы диагностики:**

1. **КОГДА** появился баг? В каком коммите/версии?
2. **ГДЕ** баг? Какие файлы/функции вовлечены?
3. **ПОЧЕМУ** происходит? Какова точная техническая причина?
4. **КАКОВО** влияние? Кто затронут? Severity?
5. **КАК** исправить? В каком слое? Breaking change?

### ФАЗА 3: ИЗОЛЯЦИЯ

Сокращай сложность, пока не найдёшь ТОЧНУЮ строку, вызывающую баг!

- **Binary search подход** — Комментируй половину кода, проверяй
- **Print debugging** — Временно! УДАЛИ после отладки!
- **Breakpoints в IDE** — Предпочтительнее print statements

### ФАЗА 4: ИСПРАВЛЕНИЕ

```markdown
## ЧЕКЛИСТ ИСПРАВЛЕНИЯ:

1. ✅ **Root cause понята** — Не гадаю, ЗНАЮ почему баг
2. ✅ **Минимальное изменение** — Фикс как можно меньше
3. ✅ **Нет побочных эффектов** — Не затрагивает другие функции
4. ✅ **Обработка краевых случаев** — Все случаи обработаны
5. ✅ **Error handling** — Try/catch, null checks где нужно
6. ✅ **Debug код удалён** — ВСЕ print/console.log убраны
7. ✅ **Код чистый** — Задокументирован, читаем
```

### ФАЗА 5: ТЕСТИРОВАНИЕ

Напиши тест, который ПАДАЕТ до фикса и ПРОХОДИТ после!

### ФАЗА 6: ВЕРИФИКАЦИЯ

1. ✅ Оригинальный баг исправлен — Не воспроизводится
2. ✅ Все тесты проходят — Нет регрессий
3. ✅ Краевые случаи покрыты
4. ✅ Производительность OK
5. ✅ OPA пройден

### ФАЗА 7: ДОКУМЕНТАЦИЯ

Commit message с: Problem → Root Cause → Solution → Testing

## 🚨 ТИПИЧНЫЕ ПАТТЕРНЫ БАГОВ:

1. **Null/Undefined References** — Используй optional chaining `?.` и `??`

## ПРОТОКОЛ 18: ROLLBACK — Откат деплоя

**Назначение:** Протокол возврата к предыдущей версии при проблемном деплое.
**Статус:** 🔒 MANDATORY для deployment issues
**Важность:** ⭐⭐⭐⭐⭐ CRITICAL

---

## 🎯 КОГДА ДЕЛАТЬ ROLLBACK:

### ✅ Триггеры ROLLBACK:
1. **Критические баги** — Новый код крашится
2. **Повреждение данных** — Новая версия портит данные
3. **Деградация производительности** — 10x+ медленнее чем было
4. **Уязвимость безопасности** — Новая брешь
5. **Высокий процент ошибок** — >5% запросов падает
6. **Нельзя быстро пофиксить** — Фикс требует часов, не минут

### ❌ НЕ откатывай (fix forward):
1. Мелкие UI баги — Можно быстро пофиксить
2. Низкий impact — Затрагивает <1% пользователей
3. Лёгкий quick fix — Hotfix за <30 мин

**Правило:** Rollback если фикс займёт >1 час ИЛИ impact крупный!

## 🔄 WORKFLOW:

### ФАЗА 1: ОЦЕНКА (2-5 минут)
- Severity? Critical/Major/Minor
- Можно ли быстро пофиксить?
- Последствия для данных?
- Последняя версия известна как хорошая?

### ФАЗА 2: ОПОВЕЩЕНИЕ (1 минута)

```
🚨 ROLLBACK В ПРОЦЕССЕ 🚨
Причина: [краткое описание]
С: v2.6.0 → На: v2.5.3
ETA: 10 минут
```

### ФАЗА 3: ROLLBACK (5-10 минут)

**Метод 1: Редеплой предыдущего тега (проще всего)**
```bash
git tag --sort=-version:refname | head -5
./deploy.sh production v2.5.3
```

**Метод 2: Git Revert (сохраняет историю)**
```bash
git revert abc123
npm run build
./deploy.sh production
```

**Метод 3: Инфраструктурный откат**
```bash
# Kubernetes
kubectl rollout undo deployment/your-app

# Heroku
heroku rollback v2.5.3 --app your-app
```

### ФАЗА 4: ВЕРИФИКАЦИЯ (3-5 минут)

1. ✅ Деплой успешен?
2. ✅ Приложение запускается?
3. ✅ Критические пути работают?
4. ✅ Процент ошибок нормальный?
5. ✅ Производительность восстановлена?

### ФАЗА 5: ОПОВЕЩЕНИЕ О ЗАВЕРШЕНИИ

```
✅ ROLLBACK ЗАВЕРШЁН ✅
Откат с v2.6.0 на v2.5.3
Статус: Production СТАБИЛЕН
```

## 🗄️ ОТКАТ БАЗЫ ДАННЫХ:

**⚠️ САМАЯ ОПАСНАЯ ЧАСТЬ!**

- **Нет изменений БД** → ✅ Безопасно, откати только код
- **Обратно-совместимая миграция** (новый столбец) → ✅ Безопасно откатить код
- **Ломающая миграция** (удалён столбец) → ⚠️ Нужен откат миграции СНАЧАЛА
- **Данные созданы после деплоя** → 🚨 Нужна оценка потери данных

## 📊 СТРАТЕГИИ ДЕПЛОЯ ДЛЯ БЫСТРОГО ОТКАТА:

1. **Blue-Green Deployment** — Два идентичных окружения, переключение трафика (<1 мин!)
2. **Canary Deployment** — Постепенный раскат (5% → 50% → 100%), минимальный impact
3. **Feature Flags** — Отключение функции без деплоя, мгновенный откат

## ⚠️ КРИТИЧЕСКИЕ ПРАВИЛА:

1. **ОЦЕНИВАЙ БЫСТРО** — Решение за <5 минут
2. **ОПОВЕЩАЙ НЕМЕДЛЕННО** — Команда в курсе
3. **ОТКАТЫВАЙ БЫСТРО** — Выполнение за <15 минут
4. **ВЕРИФИЦИРУЙ ТЩАТЕЛЬНО** — Подтверди стабильность
5. **СОХРАНЯЙ ДАННЫЕ** — Обрабатывай миграции осторожно
6. **ДОКУМЕНТИРУЙ ИНЦИДЕНТ** — Post-mortem обязателен
7. **ИСПРАВЬ ROOT CAUSE** — Не просто откати и забудь

## ЧЕКЛИСТ ВЕРИФИКАЦИИ (7 шагов)

Извлечь из одобренного ПЛАНА и проверить каждый пункт:

| Шаг | Что проверяем | Формат |
|-----|---------------|--------|
| 1 | Цели из плана | □ Цель → ✅ ВЫПОЛНЕНА / ❌ НЕ ВЫПОЛНЕНА / ⚠️ ЧАСТИЧНО + доказательство |
| 2 | Файлы CREATE из плана | □ Файл → ПЛАН: ~X lines → ФАКТ: Y lines → ✅/❌/⚠️ |
| 3 | Файлы MODIFY из плана | □ Файл → ПЛАН: [изменение] → ФАКТ: [git diff] → ✅/❌/⚠️ |
| 4 | Шаги STEP 1..N | □ STEP → ✅ ВЫПОЛНЕН / ❌ ПРОПУЩЕН / ⚠️ ИЗМЕНЁН + доказательство |
| 5 | Протоколы, объявленные "ДА" | □ Протокол → ✅ ИСПОЛЬЗОВАН / ❌ НЕ ИСПОЛЬЗОВАН |
| 6 | EXTRA файлы (незапланированные) | Список + обоснование для каждого |
| 7 | Итоговый ВЕРДИКТ | PASSED / FAILED / REQUIRES EXPLANATION |

**БЛОКИРОВКА:** Цель/файл/шаг/протокол ❌ без объяснения → СТОП!
**PASSED:** >=95% целей, >=95% файлов, >=90% шагов, 100% протоколов — выполнены ИЛИ объяснены.

## НАЗНАЧЕНИЕ

**ПРОБЛЕМА:** Во время выполнения могут появиться EXTRA файлы (для фиксов, edge cases), файлы могут быть переименованы или не созданы. Отчёт не отражает реальную структуру.
**РЕШЕНИЕ:** Обязательная верификация РЕАЛЬНЫХ файлов vs ПЛАН после выполнения.

---

## ПРОЦЕСС (4 шага)

1. **Перечислить ВСЕ реально созданные/изменённые файлы** (`git status` или `ls`)
2. **Сравнить со списком из ПЛАНА**
3. **Для КАЖДОГО расхождения документировать:**
   - `+ extra_file` (создан для: [причина])
   - `~ renamed_file` (переименован из: [оригинал] → [новое], причина)
   - `- not_created_file` (не создан для: [причина])
4. **Если структура изменилась существенно → ОБНОВИТЬ CLAUDE.md**
